# Registro de defectos de SmartSched-UC

## Criterio de registro

En este documento se registran defectos verificables en el código, la configuración, las dependencias o la documentación del repositorio SmartSched-UC.

Un defecto representa una condición concreta en la que el resultado observado difiere del resultado esperado. Cada registro incluye su causa, corrección, método de validación, estado y evidencia relacionada.

## Criterios de cierre de un defecto

Un defecto se considera cerrado cuando cumple con los siguientes criterios:

* la causa ha sido identificada;
* la corrección ha sido aplicada;
* existe evidencia verificable de la modificación;
* se ejecutó una prueba de validación;
* la corrección no produjo una regresión;
* el estado fue actualizado en este registro;
* el cambio quedó asociado con un commit o una evidencia técnica.

Cuando el defecto no puede corregirse dentro del alcance académico, debe permanecer documentado como deuda técnica o trasladarse al registro de riesgos.

## Acciones preventivas

Para evitar la repetición de defectos similares se establecen las siguientes medidas:

* conservar todos los archivos Markdown en UTF-8;
* mantener sincronizados el código y la documentación;
* no almacenar tokens ni contraseñas en archivos versionados;
* revisar el repositorio con `git grep` antes de publicar cambios;
* ejecutar `git pull` antes de fusionar ramas;
* validar los hooks de Husky en el entorno Windows;
* inspeccionar el esquema PostgreSQL después de cualquier modificación;
* ejecutar pruebas, cobertura, build y auditorías de seguridad;
* actualizar dependencias de forma gradual y controlada;
* evitar el uso de `npm audit fix --force` sin análisis previo;
* dividir componentes de gran tamaño cuando aumente su complejidad;
* relacionar cada defecto con una evidencia, riesgo, incidente o cambio.
