# Lecciones aprendidas de SmartSched-UC

## Propósito

Consolidar los aprendizajes obtenidos durante el desarrollo de SmartSched-UC, relacionando cada experiencia con su causa, impacto, acción correctiva, resultado y evidencia correspondiente.

## Síntesis de las lecciones aprendidas

* SmartSched-UC evolucionó desde un prototipo académico hacia un MVP funcional, documentado y verificable.
* La principal fortaleza del proyecto se encuentra en el motor de generación de horarios, las validaciones académicas, las pruebas del backend y la organización de evidencias.
* La implementación de PostgreSQL con fallback local permitió mantener la continuidad funcional durante las pruebas y demostraciones.
* La integración de Git, GitHub Actions, Husky, Commitlint y SonarQube fortaleció el control de calidad y la trazabilidad del proyecto.
* La cobertura del scheduler demuestra una validación sólida del componente más crítico del sistema.
* La principal oportunidad de mejora se encuentra en las pruebas del frontend y en el tratamiento de vulnerabilidades identificadas en las dependencias.
* La documentación debe actualizarse al mismo ritmo que el código para evitar diferencias entre la arquitectura descrita y la solución implementada.
* Las evidencias deben estar relacionadas con requisitos, pruebas, riesgos y cambios para facilitar la sustentación y evaluación del proyecto.
* La compilación correcta, las pruebas aprobadas y el análisis de seguridad deben interpretarse como controles diferentes y complementarios.
* La resolución de conflictos de Git demostró la importancia de mantener actualizadas las ramas y verificar el estado del repositorio antes de integrar cambios.
